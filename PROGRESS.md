  - **Real, load-bearing bugs found and fixed only by testing against the actual deployed Worker** (none of these showed up in local Node scripts or unit tests):
    - **`@aws-sdk/client-s3` doesn't work in Cloudflare Workers at all**, even just to presign a URL. `@smithy/core`'s config resolution (used internally for things like `defaultsMode`/checksum defaults) unconditionally attempts to read a shared AWS config file via `fs.readFile` as one step in its provider chain — Workers' Node-compat shim doesn't implement that, so it throws `[unenv] fs.readFile is not implemented yet!`. Tried explicitly pinning `defaultsMode`/checksum config (didn't fully fix it) and a Turbopack resolve alias to the package's own browser-safe build (didn't apply — OpenNext's server bundling is a separate esbuild pass that doesn't see Next's config). **Fix:** replaced the AWS SDK entirely with **`aws4fetch`** (SigV4 signing over `fetch`/Web Crypto only, zero Node API surface) — this is Cloudflare's own recommended approach for R2 from Workers, and it shrank the deployed bundle by ~220KB gzip as a side benefit.
    - **A crash inside `processItem` left items stuck in `"uploading"` forever**, with no retry affordance in the UI (only `"failed"` items show a Retry button). Fixed by wrapping every `processItem` call in `processQueue` with a catch-all that routes any unexpected error through the same retry/failed path.
    - **The R2 bucket had no CORS policy**, so the browser's direct PUT to the presigned URL failed with a generic `"Failed to fetch"` — invisible to any server-side log, since the request never reached our server; curl-based testing didn't catch this either, since CORS is a browser-only enforcement. Fixed via `wrangler r2 bucket cors set` (allows GET/PUT/HEAD from the app's origin).
    - **macOS's native file picker excludes files it can't classify by extension** — including a correctly-encoded HEIC with no extension at all (exactly the case `PRD.md`/`CLAUDE.md` call out), even though the picker's `accept="image/*,video/*"` should have matched it by content. Fixed by dropping the `accept` attribute on the "choose files" input entirely; we already validate every file via magic-byte sniffing server-side regardless, so nothing was gained by also letting the OS pre-filter (and doing so risked silently hiding a real photo). Drag-and-drop was never affected (it doesn't go through OS picker filtering).
  - **Scope note:** on mobile, this OS-picker-filtering issue mainly matters for browsing a phone's Files app for a specific downloaded file; capturing live (bypasses any picker) and picking from the Photos library (works off library assets, not filesystem extensions) aren't affected. The narrower Files-app case is now fixed the same way as desktop (no `accept` restriction).
  - The "compress everything before upload" behavior (not just when over the cap) and the exact `UPLOAD_MAX_DIMENSION`/quality were changed mid-milestone at the user's request, both for upload speed and to keep R2 storage within the free tier at ~100 photos/week over a year.

## Decisions

- Hosting: Cloudflare Workers via `@opennextjs/cloudflare` (Vercel Hobby is non-commercial; Pro adds $20/user/mo). `next-on-pages` ruled out — no Next 16 support.
- DB: Supabase Postgres.
- Storage: Cloudflare R2 (private bucket, signed URLs, no egress fees) — wired up and verified in Milestone 3.
- Auth: Supabase Auth, **magic-link** sign-in (no passwords to manage/reset).
- App runs on Cloudflare's `workers.dev` subdomain for now; a custom domain can be attached later without code changes.
- **No middleware/proxy for route protection.** Next.js 16's `proxy` convention (renamed from `middleware`) always compiles to the Node.js runtime, and `@opennextjs/cloudflare` 1.20.1 doesn't yet support Node-runtime proxy (Cloudflare-side error: "Node.js middleware is not currently supported"). Route protection instead lives in each protected page/layout via `requireUser()`/`requireAdmin()` — same security guarantee, just enforced per-route instead of centrally. **Trade-off to revisit:** without middleware refreshing the Supabase session cookie on every request, a signed-in user's access token (default ~1hr TTL) may expire between Server Actions/Route Handlers (the only places that can refresh it), forcing an earlier-than-ideal re-login. Not a security bug — worst case is an extra sign-in prompt. Revisit if `@opennextjs/cloudflare` adds Node-runtime proxy support, or if this becomes a real annoyance.
- **No AWS SDK anywhere in this app, by design.** `@aws-sdk/client-s3` fundamentally doesn't work in this Workers runtime (see Milestone 4's fs.readFile bug above) — R2 access goes through `aws4fetch` exclusively. Don't reintroduce the AWS SDK for any future storage work without re-testing against the deployed Worker first, not just local Node scripts.
- **R2 requires an explicit CORS policy** for any browser-direct upload/download flow (`wrangler r2 bucket cors set`) — this doesn't show up in server-side testing (curl, Node scripts) since CORS is browser-only enforcement. If a future feature adds another direct-to-R2 flow from a new origin (e.g. a custom domain), the CORS policy needs updating.
- **File `<input>` elements in this app never set `accept`** for the general "choose files" pickers — only the Capture input keeps `accept="image/*,video/*"` (needed to trigger camera mode). Don't add `accept` back to a file-picker input without checking whether it re-introduces the OS-picker-filters-by-extension bug.

## Known issues

- `wrangler` is pinned to `4.86.0` instead of latest (`4.107.0`) because the newer CLI requires Node ≥22, and this dev environment only has Node 20 available (installing Node 22 failed — sandbox has a read-only `/.cache`). This reintroduces `ws`/`undici` high-severity advisories, but only in the local deploy CLI/dev tooling, not in shipped app code or the browser bundle. **To fully resolve:** install Node 22 (e.g. via `nvm install 22` outside this sandbox) and run `npm install --save-dev wrangler@latest`.
- Session-refresh trade-off from dropping middleware — see Decisions above.
- Bootstrapping the first admin is a manual one-time step: sign up, then run `update public.users set role = 'admin' where email = '...'` in the Supabase SQL editor. No invite/user-management UI yet (that's Milestone 9).
- The upload queue survives page reloads and network drops within the same browser profile, but doesn't implement Background Sync/a service worker — a fully closed tab won't resume on its own until reopened. Acceptable for v1's guardrail (not silently losing a photo the user is actively trying to upload); full PWA background-sync hardening is a reasonable future pass, not this milestone's job.
- A retried upload after a `finalizeAsset` failure re-requests a fresh signed URL/key rather than reusing the original — the object at the old key becomes a harmless orphan in R2 (no DB row ever pointed to it). Not cleaned up automatically; negligible at this volume, but worth a periodic orphan-sweep job if storage cost ever becomes a concern.
